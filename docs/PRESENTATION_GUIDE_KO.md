# Boro Backend 발표 준비 자료

## 1. 프로젝트 한 줄 소개
**"지금 당장 필요한데 어떡하지?"** 
급할 땐 주변 이웃에게 '긴급 알림'을 보내 가장 빠르게 물건을 빌리고 빌려줄 수 있는 **동네 기반 실시간 대여/차용 플랫폼**입니다.

---

## 2. 발표 때 가장 먼저 말하면 좋은 핵심

### 아주 짧은 버전

“Boro Backend는 동네 기반 대여/차용 서비스를 위한 FastAPI 백엔드입니다. 게시글, 채팅, 거래 완료, 리뷰, 알림까지 하나의 흐름으로 연결되도록 설계했고, 실제 DB 버전과 목업 버전을 모두 지원해서 개발 초기에도 프론트엔드와 병렬로 작업할 수 있게 만들었습니다.”

### 1분 발표 버전

“이 프로젝트는 Python과 FastAPI를 사용해서 만든 지역 기반 대여 서비스 백엔드입니다.  
핵심 기능은 로그인, 사용자 정보 관리, 게시글 등록/조회, 채팅방 생성, 거래 완료 처리, 리뷰 작성, 알림 조회입니다.  
구조는 `API Router -> Service -> Repository -> Model(DB)`로 분리해서 역할을 나눴고, `MOCK_MODE`를 통해 실제 DB 없이도 화면 연동이 가능하게 만들었습니다.  
또한 PostgreSQL, SQLAlchemy, Alembic을 사용해서 데이터베이스 구조를 명확하게 관리했고, JWT 기반 인증과 예외 응답 형식도 통일해서 유지보수하기 쉽게 구성했습니다.”

---

## 3. 기술 스택과 쉬운 설명

### Python

- 이 프로젝트의 주 언어입니다.
- 문법이 비교적 읽기 쉬워서 백엔드 입문자가 배우기 좋습니다.

### FastAPI

- Python으로 API 서버를 만들기 위한 프레임워크입니다.
- 장점은 빠르고, 코드가 간결하고, Swagger 문서를 자동으로 만들어준다는 점입니다.
- 이 프로젝트에서는 모든 HTTP 요청을 FastAPI 라우터로 받습니다.

### Uvicorn

- FastAPI 앱을 실제로 실행해주는 서버입니다.
- 쉽게 말하면 “작성한 API 코드를 켜주는 프로그램”입니다.

### Pydantic

- 요청값과 응답값의 형식을 검사하고 정리해주는 도구입니다.
- 예를 들어, `price`는 숫자인지, `title`은 문자열인지 확인하는 데 도움을 줍니다.

### SQLAlchemy

- Python 코드로 데이터베이스를 다루게 해주는 ORM입니다.
- ORM은 SQL을 직접 많이 쓰지 않고, Python 객체처럼 DB 테이블을 다루게 해주는 기술입니다.
- 이 프로젝트에서는 `User`, `Post`, `ChatRoom` 같은 클래스를 DB 테이블과 연결합니다.

### PostgreSQL

- 실제 데이터를 저장하는 관계형 데이터베이스입니다.
- 사용자, 게시글, 채팅, 거래, 리뷰 같은 중요한 데이터를 저장합니다.

### Alembic

- 데이터베이스 테이블 구조 변경 이력을 관리하는 도구입니다.
- 예를 들어 “users 테이블 만들기”, “reviews 테이블 추가” 같은 변경을 기록합니다.
- 쉽게 말하면 **DB 버전 관리 도구**입니다.

### JWT

- 로그인 후 사용자를 인증하기 위한 토큰 방식입니다.
- 서버가 “이 요청이 누구의 요청인지” 확인할 때 사용합니다.

### OAuth

- 카카오 같은 외부 서비스 계정을 이용해 로그인하는 방식입니다.
- 이 프로젝트의 실제 구현은 현재 `kakao` 로그인 중심입니다.

### Docker / docker-compose

- 개발 환경을 쉽게 맞추기 위한 도구입니다.
- 이 프로젝트에서는 PostgreSQL DB를 Docker로 띄웁니다.

### httpx

- 외부 API를 호출할 때 쓰는 HTTP 클라이언트입니다.
- 여기서는 카카오 사용자 정보 API 호출에 사용됩니다.

## 4. 현재 프로젝트에서 사용한 핵심 구조

이 프로젝트의 가장 중요한 구조는 **계층 분리**입니다.

흐름은 아래와 같습니다.

1. `Router`
2. `Service`
3. `Repository`
4. `Model`
5. `Database`

쉽게 풀면:

- `Router`: URL 요청을 받는 입구
- `Service`: “이 기능을 어떻게 동작시킬지”를 결정하는 곳
- `Repository`: 실제 DB 조회/저장 담당
- `Model`: DB 테이블 모양 정의
- `Schema`: 요청/응답 데이터 형식 정의

이렇게 나누면 좋은 점:

- 코드가 읽기 쉬워집니다.
- 역할이 분명해집니다.
- 나중에 기능 수정이 쉬워집니다.
- 팀 프로젝트에서 충돌이 줄어듭니다.

---

## 5. 폴더 구조를 발표용으로 설명하는 방법

### `main.py`

- FastAPI 앱이 시작되는 진입점입니다.
- Swagger 문서 경로를 설정하고, 전체 라우터와 예외 핸들러를 등록합니다.

### `app/api/v1`

- 실제 API 엔드포인트가 들어 있습니다.
- 예: `/api/posts`, `/api/users/me`
- 발표에서는 “컨트롤러 역할”이라고 설명해도 됩니다.

### `app/core`

- 공통 설정과 핵심 동작이 들어 있습니다.
- 환경변수 로딩, DB 연결, 의존성 주입, 예외 처리 등을 담당합니다.

### `app/models`

- 데이터베이스 테이블 구조를 Python 클래스로 정의합니다.

### `app/schemas`

- 요청 바디와 응답 데이터 형식을 정의합니다.
- 프론트엔드와 약속하는 데이터 형식이라고 생각하면 됩니다.

### `app/repositories`

- DB 접근만 담당합니다.
- 예를 들어 게시글 검색, 사용자 조회, 리뷰 저장 등을 수행합니다.

### `app/services/impl`

- 실제 비즈니스 로직이 들어 있습니다.
- 예: 글 작성 조건 검사, 거래 완료 시 상태 변경, 리뷰 작성 시 평점 반영

### `app/services/mock`

- 가짜 데이터를 반환하는 목업 서비스입니다.
- 프론트엔드가 먼저 붙을 수 있도록 만든 임시/개발용 구현입니다.

### `alembic`

- DB 마이그레이션 파일이 들어 있는 폴더입니다.

### `docs`

- 프로젝트 문서 폴더입니다.

## 6. 전체 요청 흐름

예를 들어 사용자가 게시글 상세 조회를 요청한다고 가정하면:

1. 클라이언트가 `GET /api/posts/{post_id}` 요청을 보냅니다.
2. FastAPI Router가 요청을 받습니다.
3. `deps.py`에서 현재 설정을 보고 mock 서비스 또는 실제 서비스 중 하나를 주입합니다.
4. `PostsService`가 게시글 존재 여부, 사용자 좋아요 여부 등을 확인합니다.
5. `PostRepository`가 DB에서 게시글과 작성자, 이미지 정보를 조회합니다.
6. 결과를 `schema` 형식에 맞춰 응답합니다.

이 구조를 발표에서 말하면 “단순히 API만 만든 게 아니라, 유지보수를 고려한 구조로 설계했다”는 인상을 줄 수 있습니다.

## 7. 기술적으로 중요한 포인트

### 7-1. `MOCK_MODE` 지원

이 프로젝트의 가장 발표하기 좋은 특징 중 하나입니다.

- `MOCK_MODE=true`이면 실제 DB를 쓰지 않고 mock 데이터를 반환합니다.
- `MOCK_MODE=false`이면 실제 DB와 서비스 로직을 사용합니다.

왜 좋은가:

- 프론트엔드가 백엔드 완성 전에도 화면 개발 가능
- API 명세를 먼저 맞출 수 있음
- 실제 DB가 아직 준비 안 된 상황에서도 시연 가능

발표 때 한 줄 요약:

“개발 초기 생산성을 높이기 위해 mock 서비스와 실제 서비스를 분리했고, 환경변수만 바꾸면 실제 DB 모드로 전환할 수 있게 만들었습니다.”

### 7-2. 의존성 주입(Dependency Injection)

`app/core/deps.py`가 서비스 객체를 선택해서 넣어줍니다.

쉽게 설명하면:

- Router가 직접 서비스를 만들지 않습니다.
- 공통 함수가 대신 만들어줍니다.
- 그래서 mock과 실제 구현을 쉽게 바꿀 수 있습니다.

### 7-3. 공통 응답 형식

성공 응답은 대부분 아래 형태입니다.

```json
{
  "data": { }
}
```

장점:

- 프론트엔드가 응답 구조를 예측하기 쉬움
- API 문서가 일관됨

### 7-4. 공통 예외 처리

에러는 `AppError`를 기반으로 통일된 형식으로 반환합니다.

쉽게 말해:

- 어떤 API에서 에러가 나도 응답 모양이 비슷합니다.
- 프론트엔드가 에러를 처리하기 쉬워집니다.

### 7-5. 페이지네이션(Pagination)

게시글, 거래, 리뷰, 알림 등에서 `page`, `size`, `has_next`를 사용합니다.

페이지네이션은 데이터를 한 번에 다 주지 않고 나눠서 주는 방식입니다.  
데이터가 많아질수록 꼭 필요한 설계입니다.

### 7-6. DB 제약조건 활용

이 프로젝트는 중복을 막기 위해 DB 차원 제약조건도 사용합니다.

예:

- 같은 사용자가 같은 게시글에 좋아요를 여러 번 누르지 못하게 함
- 같은 채팅방에서 거래를 여러 번 완료하지 못하게 함
- 한 거래에 대해 같은 사용자가 리뷰를 두 번 쓰지 못하게 함

이건 발표에서 “비즈니스 규칙을 코드뿐 아니라 DB에서도 보호했다”고 설명하면 좋습니다.

## 8. 구현된 기능 정리

전체 API는 크게 6개 영역이고, 엔드포인트 수는 총 31개입니다.

### 8-1. 인증(Auth)

기능:

- 소셜 로그인
- 토큰 재발급
- 로그아웃
- 회원 탈퇴

구현 포인트:

- 실제 구현은 카카오 OAuth 로그인 중심
- 로그인 성공 시 access token / refresh token 발급
- JWT payload에 `sub`, `type`, `exp`를 넣어 토큰 구분
- 현재 `logout()`은 서버에서 블랙리스트를 두는 방식은 아니고, stateless JWT 구조라 클라이언트에서 토큰 폐기 개념으로 처리

발표 포인트:

“토큰 기반 인증 구조를 도입했고, refresh token을 분리해서 재로그인 없이 access token을 갱신할 수 있게 했습니다.”

### 8-2. 사용자(Users)

기능:

- 내 프로필 조회
- 내 프로필 수정
- 위치 갱신
- 설정 수정
- 내가 쓴 글 조회
- 내가 좋아요한 글 조회
- 다른 사용자 프로필 조회
- 다른 사용자 리뷰 조회

구현 포인트:

- 사용자의 거래 횟수와 좋아요 수를 별도로 계산
- 위치는 위도/경도를 받아 지역명으로 변환하는 흐름을 고려
- 현재 실제 주소 변환은 TODO 상태이며, 예시 주소를 사용
- 관심 키워드는 중복 없이 저장되도록 정리

발표 포인트:

“단순히 사용자 테이블만 조회하는 것이 아니라, 거래 수, 좋아요 수, 리뷰 수 같은 파생 정보도 함께 제공하도록 설계했습니다.”

### 8-3. 게시글(Posts)

기능:

- 게시글 목록 조회
- 게시글 작성
- 게시글 상세 조회
- 게시글 수정
- 게시글 삭제
- 좋아요 등록/해제
- 게시글에서 채팅방 생성

구현 포인트:

- 키워드, 글 종류, 카테고리, 긴급 여부, 지역명으로 필터링 가능
- 정렬 기준은 `created_at`, `price`, `like_count`
- 게시글 작성/수정 시 입력값 검증 수행
- 게시글 삭제 시 이미 채팅이나 거래가 있으면 삭제를 막음
- 이미지 URL 목록을 별도 테이블(`post_images`)로 관리

발표 포인트:

“게시글은 단순 CRUD가 아니라 검색, 필터링, 이미지 관리, 좋아요, 채팅 생성까지 연결되는 중심 기능입니다.”

### 8-4. 채팅(Chats)

기능:

- 채팅방 목록 조회
- 채팅방 상세 조회
- 메시지 목록 조회
- 메시지 전송
- 읽음 처리

구현 포인트:

- 게시글 기준으로 채팅방 생성
- 참여자 역할을 `lender`, `borrower`로 저장
- 마지막 메시지 미리보기와 마지막 메시지 시간 저장
- 안 읽은 메시지 수(`unread_count`) 관리
- 메시지 조회는 cursor 기반에 가까운 방식 사용

중요한 설명:

이 채팅은 **REST API 기반 채팅**입니다.  
즉, 웹소켓(WebSocket) 기반 실시간 채팅은 아직 아닙니다.

교수님 질문 대비:

“실시간 소켓까지는 구현하지 않았고, 우선 REST 구조로 채팅 데이터 모델과 읽음 처리 흐름을 먼저 안정적으로 설계했습니다.”

### 8-5. 거래(Transactions)와 리뷰(Reviews)

기능:

- 거래 목록 조회
- 거래 완료 생성
- 거래 상세 조회
- 리뷰 작성

구현 포인트:

- 거래 완료 시 게시글 상태를 `COMPLETED`로 변경
- 대여자/차용자 역할을 게시글 종류(`LEND`, `BORROW`)에 따라 자동 결정
- 두 사용자 모두 `completed_transaction_count` 증가
- 리뷰 작성 시 상대방의 평균 평점을 다시 계산해 `trust_score`에 반영

발표 포인트:

“거래 완료는 단순 insert가 아니라 게시글 상태와 사용자 신뢰도에 영향을 주는 핵심 비즈니스 로직입니다.”

### 8-6. 알림(Notifications)

기능:

- 알림 목록 조회
- 읽음 처리

알림 타입:

- `urgent_post`
- `interest_post`
- `chat_message`

설명 포인트:

- 알림 생성 로직까지 완전히 자동화한 단계는 아니고
- 우선 알림 데이터 구조와 조회/읽음 처리 API를 설계한 상태입니다.

## 9. 데이터베이스 설계 핵심

이 프로젝트는 아래 테이블들이 핵심입니다.

### `users`

- 사용자 기본 정보
- 닉네임, 프로필 이미지, 지역, 신뢰 점수 저장

### `social_accounts`

- 소셜 로그인 연결 정보
- 어떤 플랫폼 계정으로 로그인했는지 저장

### `posts`

- 대여/차용 게시글 본문 저장

### `post_images`

- 게시글에 연결된 이미지 저장

### `post_likes`

- 어떤 사용자가 어떤 게시글을 좋아요했는지 저장

### `chat_rooms`

- 게시글 단위로 생성된 채팅방 저장

### `chat_room_participants`

- 채팅방 참여자 목록과 역할, 읽음 상태 저장

### `chat_messages`

- 실제 채팅 메시지 저장

### `transactions`

- 거래 완료 기록 저장

### `reviews`

- 거래 후 사용자 간 리뷰 저장

### `notifications`

- 사용자에게 보여줄 알림 저장

발표에서는 이렇게 설명하면 좋습니다.

“ERD를 보면 게시글을 중심으로 채팅과 거래가 연결되고, 거래가 끝나면 리뷰로 이어지는 흐름을 볼 수 있습니다. 즉, 데이터 구조 자체가 서비스 사용자 흐름을 반영하도록 설계했습니다.”

### 장점 1. 구조 분리가 깔끔함

### 장점 3. 비즈니스 규칙이 서비스 계층에 잘 모여 있음

- 거래 완료 시 상태 변경

### 장점 4. 데이터 무결성에 신경 씀

## 11. 아쉬운 점과 개선 방향

발표에서는 아쉬운 점을 숨기기보다, “어디까지 했고 무엇을 다음 단계로 남겨뒀는지”를 솔직하게 말하는 편이 좋습니다.

### 현재 한계

1. 기본 실행이 `MOCK_MODE=true`라서 실제 운영형 구조는 준비되어 있지만 기본 시연은 mock 중심입니다.
2. 실제 OAuth 구현은 카카오 위주이며, 구글은 환경변수만 있고 구현은 보이지 않습니다.
3. `logout()`은 서버 블랙리스트 방식이 아니라 stateless JWT 구조입니다.
4. 위치 업데이트는 실제 역지오코딩 API 연동 대신 예시 주소를 사용합니다.
5. 알림 생성 자동화 로직은 아직 없습니다.
6. 채팅은 웹소켓 기반 실시간 구조가 아니라 REST 기반입니다.
7. 테스트 코드가 없습니다.
8. 일부 코드 스타일은 통일감이 조금 부족합니다.
   예를 들어 어떤 Repository는 SQLAlchemy `select()` 스타일을 쓰고, 어떤 곳은 `db.query()` 스타일을 씁니다.

### 개선 방향

1. WebSocket 기반 실시간 채팅 도입
2. Redis를 이용한 알림/메시지 처리 개선
3. 카카오 외 구글 로그인 추가
4. 실제 주소 변환 API 연동
5. pytest 기반 테스트 코드 작성
6. CI/CD 파이프라인 연결
7. 코드 스타일 통일 및 리팩터링

## 12. 초보자용 IT 용어 사전

### API

프로그램끼리 대화하기 위한 약속입니다.  
예를 들어 프론트엔드가 “게시글 목록 줘”라고 백엔드에 요청하는 통로입니다.

### Endpoint

API의 실제 주소입니다.  
예: `/api/posts`

### Router

어떤 주소 요청이 들어왔을 때 어떤 함수를 실행할지 연결하는 곳입니다.

### Service

“이 기능이 실제로 어떻게 동작해야 하는지”를 적는 곳입니다.  
비즈니스 로직이 모이는 곳입니다.

### Repository

DB와 직접 대화하는 계층입니다.

### Model

DB 테이블 구조를 코드로 표현한 클래스입니다.

### Schema

요청과 응답 데이터 형식을 정의한 클래스입니다.

### ORM

데이터베이스를 SQL 문장 대신 객체처럼 다루게 해주는 기술입니다.  
이 프로젝트에서는 SQLAlchemy가 ORM 역할을 합니다.

### Migration

DB 구조 변경 이력을 관리하는 작업입니다.  
Alembic이 이를 담당합니다.

### JWT

로그인 후 사용자 인증 정보를 담는 토큰입니다.

### OAuth

카카오, 구글 같은 외부 계정으로 로그인하는 방식입니다.

### Dependency Injection

필요한 객체를 직접 만들지 않고 외부에서 넣어주는 방식입니다.  
이 프로젝트에서는 mock 서비스와 실제 서비스를 쉽게 바꾸기 위해 사용합니다.

### Pagination

데이터를 여러 페이지로 나눠서 조회하는 방식입니다.

### Mock

실제 DB나 외부 API 대신, 미리 만들어둔 가짜 응답 데이터를 뜻합니다.

### Validation

입력값이 올바른지 검사하는 과정입니다.

### Constraint

DB에서 지켜야 하는 규칙입니다.  
예: 같은 좋아요를 두 번 누를 수 없음

### Stateless

서버가 로그인 상태를 따로 저장하지 않고, 토큰만 보고 인증하는 구조입니다.

## 13. 교수님 예상 질문과 답변

### Q1. 왜 FastAPI를 선택했나요?

A. Python 기반에서 API 서버를 빠르게 만들 수 있고, Swagger 문서가 자동 생성되어 개발과 테스트가 편하기 때문입니다. 입문 프로젝트이면서도 구조화된 백엔드를 만들기에 적합하다고 판단했습니다.

### Q2. 왜 Service와 Repository를 분리했나요?

A. 역할을 나누기 위해서입니다. Service는 비즈니스 규칙을 담당하고, Repository는 DB 접근만 담당하게 해서 유지보수성과 가독성을 높였습니다.

### Q3. `MOCK_MODE`를 왜 만들었나요?

A. 프론트엔드와 백엔드가 동시에 개발될 때, 실제 DB나 외부 로그인 구현이 완성되지 않아도 화면 연동과 API 테스트를 진행할 수 있도록 하기 위해서입니다.

### Q4. 현재 이 프로젝트는 실제 서비스 수준인가요?

A. 구조적으로는 실제 서비스 형태를 지향하고 있지만, 현재는 mock 모드가 기본이고 테스트 코드, 실시간 채팅, 실제 주소 API 연동, 알림 자동 생성 같은 부분은 추가 개발이 필요합니다.

### Q5. 왜 PostgreSQL을 사용했나요?

A. 사용자, 게시글, 거래, 리뷰처럼 관계가 뚜렷한 데이터가 많아서 관계형 데이터베이스가 적합하다고 판단했습니다.

### Q6. 채팅은 왜 WebSocket이 아니라 REST인가요?

A. 우선 데이터 구조와 기능 흐름을 먼저 안정적으로 설계하는 데 집중했습니다. 이후 확장 단계에서 WebSocket으로 발전시키기 쉬운 구조를 목표로 했습니다.

### Q7. 거래 완료 시 왜 게시글 상태를 바꾸나요?

A. 이미 거래가 끝난 게시글에 대해 추가 채팅이나 중복 거래가 생기지 않도록 상태를 명확히 관리하기 위해서입니다.

### Q8. 신뢰 점수는 어떻게 계산되나요?

A. 리뷰가 작성될 때 해당 사용자가 받은 리뷰 평점 평균을 다시 계산해서 `trust_score`에 반영합니다.

### Q9. 좋아요 수와 채팅 수를 왜 테이블에서 매번 세지 않고 컬럼으로 들고 있나요?

A. 목록 조회 성능을 고려한 캐시 성격의 컬럼으로 볼 수 있습니다. 자주 보여주는 값이라 미리 저장해두면 조회가 단순해집니다.

### Q10. 인증은 어떻게 처리하나요?

A. access token과 refresh token을 발급하는 JWT 기반 인증입니다. 요청 시 access token으로 사용자 정보를 확인합니다.

### Q11. 왜 예외 응답 형식을 통일했나요?

A. 프론트엔드가 에러를 일관되게 처리할 수 있고, 디버깅과 문서화가 쉬워지기 때문입니다.

### Q12. 리뷰를 한 사람이 여러 번 쓰지 못하도록 어떻게 막았나요?

A. 서비스 로직에서 한 번 검사하고, DB unique 제약조건으로 한 번 더 막았습니다. 즉, 애플리케이션과 DB 양쪽에서 중복을 방지했습니다.

### Q13. 가장 중요한 설계 포인트 하나만 꼽는다면?

A. `MOCK_MODE`를 포함한 계층 분리 구조입니다. 개발 속도와 유지보수성을 둘 다 챙길 수 있게 해준 핵심 설계라고 생각합니다.

### Q14. 아쉬운 점은 무엇인가요?

A. 테스트 코드 부재와 일부 미구현 기능입니다. 다만 구조를 먼저 안정적으로 만든 뒤 세부 기능을 확장하는 방향으로 개발했습니다.

### Q15. 만약 다음 단계로 발전시킨다면 무엇을 먼저 하겠나요?

A. 실시간 채팅(WebSocket), 테스트 코드 작성, 실제 주소 API 연동, 알림 자동 생성 기능을 우선순위로 두겠습니다.
